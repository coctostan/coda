## Why This Milestone

The v0.9 Live Compounding Validation milestone (Phase 52) proved that v0.8's compounding engine has a complete set of plumbing (655 tests, TypeScript clean) but **does not compound in practice**. When a real agent walked through the lifecycle:

- Agent couldn't enter the lifecycle autonomously (no agent-callable forge/focus)
- UNIFY produced zero overlays and zero reference doc updates
- Supporting systems (coda_run_tests, write gate) had regressions
- Agent's "just build it" default overrode the lifecycle prompts

v0.10 closes the agent loop — makes the compounding engine actually compound when an agent uses it, end-to-end.

## Theme

**"The compounding engine actually compounds."**

Every fix is tied directly to a finding from `docs/v0.8/E2E-COMPOUNDING-FINDINGS.md`. The success criterion is a clean re-run of Script A where overlays appear, reference docs update, and Issue 2 measurably benefits from Issue 1's compounded knowledge.

## Phases

### Phase 53: Agent Entry Points
**Addresses:** F1 (forge not agent-callable), F2 (no focus tool)
**Goal:** Make FORGE and focus-issue agent-callable, not just slash commands.

- Add `coda_forge` tool — agent-callable wrapper around the forge flow
- Add `coda_focus` tool — agent-callable issue activation (slug → focus_issue + phase + branch)
- Keep slash commands as convenience wrappers calling the tools
- Reinstate write gate to block direct `.coda/` edits (not just creation) — fixes F7 regression

**Testable outcome:** Agent in a bare workspace: `coda_forge` → `coda_create(issue)` → `coda_focus(slug)` without touching slash commands or state.json.

### Phase 54: UNIFY Actually Produces Artifacts
**Addresses:** F5 (UNIFY produces zero artifacts) — **the headline fix**
**Goal:** Make UNIFY generate overlays and reference doc updates, not just completion records.

- Investigate why only action 5 (completion record) runs
- Restructure UNIFY runner prompt: each of 5 mandatory actions gets a clear name, specific tool call, output path, runner checkpoint
- Consider iterative/gated UNIFY: completion record not accepted until overlays+ref docs exist on disk
- Add post-UNIFY validation: `unify→done` blocked unless ≥1 overlay and ≥1 ref-doc update (or explicit exemption)

**Testable outcome:** Re-run of Script A's UNIFY produces ≥1 non-empty overlay + ref-system.md update. Unit test: UNIFY can't advance to `done` without artifacts.

### Phase 55: Supporting Systems Repair
**Addresses:** F6 (coda_run_tests broken), F7 (write gate regression), deferred item #1
**Goal:** Fix the surrounding systems that broke during E2E.

- Fix `coda_run_tests` "Bun is not defined" — resolve bun from PATH, handle spawn context
- Strengthen write gate: block *all* direct `.coda/` writes (creation AND edits), fail-closed
- Remove `human_review_default` from CodaConfig (deferred item #1) — auto-migrate legacy configs to `gates.*_review: "human"` on read

**Testable outcome:** `coda_run_tests` runs cleanly under Pi. Direct `write` to `.coda/state.json` blocked. Legacy `human_review_default` configs still work via migration.

### Phase 56: Lifecycle-First Prompts
**Addresses:** F3 (agent builds first), F4 (agent reads source to understand tools)
**Goal:** Make the agent naturally follow CODA's lifecycle instead of bypassing it.

- Audit all CODA-injected prompts for lifecycle guidance
- Strengthen session-start prompt: explicit "issue first before building", compact tool manifest, 1-paragraph lifecycle overview
- Inline examples: "user says 'add X' → call coda_create({...}) then coda_focus({slug: '...'})"
- "Next expected action" hint in every `coda_status` response tailored to current state

**Testable outcome:** Fresh agent, fresh project, told "I want to add X" — creates CODA issue within first 2 tool calls without nudging.

### Phase 57: E2E Re-Validation
**Addresses:** All findings — validates the milestone
**Goal:** Prove the compounding engine actually compounds.

- Re-run Script A (greenfield URL shortener → analytics) with v0.10 code
- Success criteria:
  - [ ] Agent enters lifecycle without manual `state.json` edits
  - [ ] Issue 1 UNIFY produces ≥1 overlay with non-boilerplate content
  - [ ] Issue 1 UNIFY updates ref-system.md with actual capabilities
  - [ ] Issue 2 shows measurable benefit (references Issue 1 patterns, uses overlay guidance, fewer mistakes)
  - [ ] No slash commands needed after `/coda` initial load
  - [ ] TDD gate unlocks properly via `coda_run_tests`
- Write findings to `docs/v0.10/E2E-COMPOUNDING-FINDINGS-v2.md`
- If compounding STILL doesn't work, surface that clearly — don't paper over it
- Optional stretch: Script B (brownfield) if Script A passes with time to spare

**Testable outcome:** Findings doc declares the compounding hypothesis validated OR explicitly names what's still broken.

## Success Definition

v0.10 is complete when:
1. All 7 critical/high findings from v0.8 E2E have code fixes with tests
2. A live CMUX run of Script A produces overlays, updates ref-system.md, and shows Issue 2 benefiting from Issue 1
3. The agent navigates the full lifecycle autonomously — no slash commands after session start, no manual state edits

**If Phase 57 re-validation fails to show compounding, the milestone is NOT complete, regardless of test count. Don't move the goalposts.**

## Risks / Open Questions

1. **Phase 54 risk:** What if UNIFY problem isn't just prompt quality? What if the agent fundamentally needs more structured instructions? **Fallback:** make UNIFY phase-locked per action (each action becomes a mini-phase with its own gate).

2. **Phase 56 risk:** Strengthening prompts might bloat context budgets. Need to measure token impact and stay within v0.7 budgets.

3. **Phase 57 risk:** What if even with fixes, GPT-5.4's "just build it" default is too strong to override? **Fallback:** document this as a fundamental limitation and design around it ("CODA requires a planning-aware model" becomes part of the PRD).

4. **Phase 53 open question:** Should `coda_focus` also create the VCS branch, or separate? Current `/coda activate` does both. Probably keep together for now, revisit if it causes issues.
