## Why This Milestone

Phase 57 closed v0.10 as a documented failure. The compounding infrastructure is real, and the entry surface improved, but the live Script A rerun still failed at the points that actually determine whether CODA compounds in practice: SPECIFY → PLAN advancement, explicit VERIFY evidence, and reachable UNIFY artifact production.

v0.11 is therefore a repair-and-proof milestone, not a feature-expansion milestone. The job is to fix the observed live failures, then re-run Script A without changing the success criteria.

## Theme

**"Fix the live lifecycle gaps, then prove compounding works."**

## Live Failure Map

| # | Finding | Root Cause Summary | Scope |
|---|---------|--------------------|-------|
| C1 | `coda_advance` crashes SPECIFY→PLAN | `coda-advance.ts` performs nil-unsafe frontmatter reads in a live path; defensive handling is incomplete | Code + tests |

## Critical Framing Adjustments

- **Do not re-solve Phase 54 abstractly.** The evidence gate already verifies declared artifacts on disk. v0.11 must prove that the reachable lifecycle actually gets to a truthful UNIFY and satisfies that gate.
- **Test-command defaults must stay conservative.** If the toolchain is inferable, seed a real default. If it is not, record an explicit follow-up rather than guessing wrong.
- **VERIFY enforcement should focus on evidence, not mind-reading.** We can require explicit suite results and failed live checks to block success claims. We cannot mechanically guarantee perfect interpretation of every runtime probe.

## Phases

### Phase 58: Lifecycle Bug Fixes

**Addresses:** C1, C2, C3
**Goal:** Remove the concrete code-path bugs that blocked or degraded the live v0.10 rerun.

- Fix nil-safe frontmatter handling in `coda-advance.ts` for SPECIFY → PLAN and do a defensive sweep for similar optional-field reads.
- Fix `coda_edit_body` section replacement semantics so semantically-matching headings do not silently duplicate.
- Improve scaffold defaults for test commands: infer Bun/npm commands when the project shape makes that safe; otherwise leave an explicit required-follow-up signal instead of silent null ambiguity.

**Exit criteria:**
- [ ] `coda_advance` no longer crashes on missing/partial frontmatter in the live SPECIFY → PLAN path
- [ ] `coda_edit_body` replace-section flows do not create duplicate headings for equivalent sections
- [ ] Scaffolded `coda.json` contains sensible test defaults when inferable, otherwise a clear operator follow-up
- [ ] Existing repo test baseline stays green

### Phase 59: Lifecycle Integrity

**Addresses:** C4, C5
**Goal:** Make the reachable lifecycle trustworthy end-to-end once Phase 58 unblocks it.

- Make VERIFY fail closed on missing verification evidence.
- Strengthen VERIFY guidance so the agent must run `coda_run_tests` and report live-check failures when acceptance criteria require runtime behavior.
- Restructure the UNIFY prompt into explicit, sequential actions so artifact-producing steps are harder to skip.
- Preserve and exercise the existing Phase 54 evidence gate in reachable lifecycle tests.
- If prompt-only sequencing still fails in bounded tests, escalate within this milestone to a more structured UNIFY execution mechanism rather than pretending the prompt is sufficient.

**Exit criteria:**
- [ ] VERIFY rejects advancement without explicit verification evidence
- [ ] Failed live checks cannot coexist with a successful VERIFY claim in the covered test path
- [ ] UNIFY produces and declares real artifact paths in a reachable end-to-end test
- [ ] Programmatic lifecycle coverage proves SPECIFY → PLAN → BUILD → VERIFY → UNIFY → DONE can complete honestly

### Phase 60: E2E Re-Validation

**Addresses:** C6 and the milestone proof obligation
**Goal:** Re-run Script A and decide, with no goalpost movement, whether compounding now works in practice.

- Use a fresh external workspace and the same Script A source of truth.
- Re-check the exact failure modes from Phase 57.
- Record a binary outcome: validated or still broken.

**Hard success criteria:**
- [ ] Agent enters via `coda_forge` → `coda_create` → `coda_focus`
- [ ] `coda_advance` SPECIFY → PLAN succeeds in the live run
- [ ] TDD gate works without manual config surgery
- [ ] VERIFY includes explicit evidence and reports failed live checks honestly
- [ ] UNIFY produces at least one non-boilerplate overlay and a reference-doc update
- [ ] Issue 2 shows real artifact-mediated carry-forward
- [ ] No duplicate `.coda/` section headings appear in the exercised path

## Deferred / Out of Scope

- Brownfield Script B re-validation — still deferred until greenfield compounding works live
- New FORGE capabilities or module expansion
- General refactor work not required to clear the live failures (for example, the `coda-advance.ts` soft-size overage)
- README / milestone-close doc sweep unless it becomes necessary to land one of the repair phases

## Success Definition

v0.11 is complete when:
1. The six live findings from Phase 57 have either been fixed or collapsed into fewer, better-explained failure modes
2. Script A is rerun against shipped v0.11 behavior with a binary, evidence-backed verdict
3. If the run still fails, the milestone says so explicitly instead of crediting partial improvements as compounding

## Risks / Fallbacks

1. **UNIFY may still skip steps even after prompt restructuring.** If bounded tests show that, escalate to a more structured execution model inside Phase 59.
2. **VERIFY honesty is only partly enforceable.** The hard requirement is explicit evidence capture and failed-check propagation.
3. **Toolchain detection can be wrong in sparse projects.** Prefer conservative defaults and explicit follow-up over false certainty.
