# Phase 50, Plan 01: Gate Automation

## Objective

Replace the boolean `human_review_default` and hard-coded gate behavior with a configurable gate automation system. Each lifecycle gate point becomes configurable as `human`, `auto`, or `auto-unless-block`, with project-level defaults and per-issue-type overrides in `coda.json`.

## Design

### Gate points (from spec)

| Gate Name | Controls | Default Mode | Hardcoded Fallback |
|-----------|----------|-------------|-------------------|
| `plan_review` | PLAN â†’ BUILD human review requirement | `human` | `human` |
| `build_review` | BUILD â†’ VERIFY module findings block | `auto-unless-block` | `auto-unless-block` |
| `unify_review` | UNIFY â†’ DONE human approval | `human` | `human` |

**Scoped for v0.8:** Only these 3 gate points are configurable. They correspond to the 3 gates that currently have non-structural checks (human approval or module-block behavior). The structural gates (`specifyâ†’plan`: AC count, `planâ†’review`: plan exists, `verifyâ†’unify`: ACs met) remain hard-coded â€” they check data completeness, not approval policy.

**Rationale for scoping:** The spec lists 6 gate points including `specify_approval`, `spec_delta_review`, and `reference_doc_updates`. However, `specify_approval` has no current implementation (SPECIFYâ†’PLAN only checks AC count), and `spec_delta_review`/`reference_doc_updates` are UNIFY sub-actions (not phase transitions). Implementing those would require new gate infrastructure beyond phase transitions. Phase 50 focuses on the 3 phase-transition approval gates that already have working mechanics.

### Mode semantics
- **`human`** â€” gate requires explicit human approval (current default for plan_review, unify_review)
- **`auto`** â€” gate passes automatically (no human interaction needed)
- **`auto-unless-block`** â€” passes automatically unless module findings exceed threshold (current default for build_review)

### Resolution order
1. `config.gate_overrides?.[issueType]?.[gateName]` â€” per-issue-type override
2. `config.gates?.[gateName]` â€” project default
3. Hardcoded fallback per gate

### Backward compatibility
- If `gates` field is absent but `human_review_default` is present, migrate at read time:
  - `human_review_default.feature === true` â†’ `plan_review: 'human'` for features
  - `human_review_default.feature === false` â†’ `plan_review: 'auto'` for features
- If both `gates` and `human_review_default` are present, `gates` wins
- `human_review_default` stays in `CodaConfig` type as optional/deprecated

## Deliverables

### D1: Gate Automation Types (`packages/coda/src/workflow/gate-automation.ts`) â€” NEW

**Types:**
```typescript
type GateMode = 'human' | 'auto' | 'auto-unless-block';

type GatePoint = 'plan_review' | 'build_review' | 'unify_review';

interface GateAutomation {
  plan_review: GateMode;
  build_review: GateMode;
  unify_review: GateMode;
}

interface GateOverrides {
  [issueType: string]: Partial<GateAutomation>;
}
```

**Functions:**
- `resolveGateMode(gate: GatePoint, issueType: string, config: CodaConfig): GateMode`
  - Checks `config.gate_overrides?.[issueType]?.[gate]` first
  - Falls back to `config.gates?.[gate]`
  - Falls back to `DEFAULT_GATE_MODES[gate]`
  - Handles backward compat: if no `gates` but `human_review_default` present, derives `plan_review` mode
- `shouldRequireHuman(mode: GateMode, hasBlockingFindings: boolean): boolean`
  - `human` â†’ `true`
  - `auto` â†’ `false`
  - `auto-unless-block` â†’ `hasBlockingFindings`
- `DEFAULT_GATE_MODES: Record<GatePoint, GateMode>`
  - `plan_review: 'human'`
  - `build_review: 'auto-unless-block'`
  - `unify_review: 'human'`
