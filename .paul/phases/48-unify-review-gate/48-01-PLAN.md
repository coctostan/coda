## Objective

Add a human review/approval mechanism for UNIFY output so the human validates ref-system.md changes, completion record content, and reference doc updates before the agent can advance to DONE. This closes the loop on D1 (UNIFY remains autonomous; human validation via gate-mediated review step).

## Design

### Pattern Reuse

Phase 48 mirrors the existing **review→build** human review gate from Phase 12:

| Concept | review→build (Phase 12) | unify→done (Phase 48) |
|---------|------------------------|----------------------|
| Trigger field | `issue.human_review: true` | Always required (v0.8 default) |
| Status field | `plan.human_review_status` | `completionRecord.unify_review_status` |
| Gate check | `humanReviewRequired && status !== 'approved'` | `unifyReviewStatus !== 'approved'` |
| Approve path | `AdvanceInput.human_review_decision: 'approved'` | `AdvanceInput.unify_review_decision: 'approved'` |
| Reject path | Changes-requested → revise submode | Changes-requested → capture feedback on completion record, stay in unify |
| Revision context | `loadRevisionInstructions()` in review runner | `loadUnifyReviewFeedback()` in unify runner |
| Status display | `StatusResult.human_review_status` | `StatusResult.unify_review_status` |
| Command UX | `/coda advance` (auto-detects pending) | `/coda advance` (auto-detects pending) |

### Key Design Decisions

1. **UNIFY review is always required in v0.8** — No per-issue toggle. Phase 50 (Gate Automation) will add configurability later.
2. **Completion record carries the review status** — New `unify_review_status` field on CompletionRecord (`'pending' | 'approved' | 'changes-requested'`).
3. **Changes-requested captures feedback on the completion record** — Via `coda_edit_body` `replace_section` "UNIFY Review" on the completion record, so only the latest feedback is visible. Agent stays in UNIFY phase to address feedback.
4. **No new submode** — Unlike review→build which uses `revise` submode, UNIFY review stays in `unify` phase. The completion record's `unify_review_status: 'changes-requested'` is the disk-based signal that distinguishes a revision pass from a first pass.
5. **UNIFY runner detects revision state from disk** — Since the spec mandates "fresh autonomous session, reads everything from disk" for UNIFY, `assembleUnifyContext()` must detect an existing completion record with `unify_review_status: 'changes-requested'`, load the "UNIFY Review" feedback section, and provide a revision-aware prompt instead of the vanilla 5-action prompt.
6. **Aggregate review gate, not per-action approval** — The spec describes per-edit human approval during UNIFY Actions 1 and 3 ("Extension previews the diff to the human. Human approves or rejects."). For v0.8, diffs are visible in the conversation channel (D2) and the review gate at `unify→done` serves as the aggregate "are you satisfied with everything?" checkpoint. Per-action interrupt-based approval is deferred as a future enhancement.
7. **Shared `findCompletionRecordPath()` helper** — Three call sites need to locate the completion record: `gatherGateData()`, `handleUnifyReviewDecision()`, and `assembleUnifyContext()`. A shared helper prevents divergent duplication.

### Flow

```
Agent performs 5 UNIFY actions autonomously
  → Agent writes completion record with unify_review_status: 'pending'
  → Agent calls coda_advance(target_phase: 'done')
  → Gate blocks: "UNIFY review pending — human must approve"
  → coda_status shows: "UNIFY review pending — approve to finalize or request changes"

Happy path:
  → Human: /coda advance
    → sets unify_review_status to 'approved'
    → gate passes → DONE

Changes-requested path:
  → Human: /coda advance changes <feedback>
    → sets unify_review_status to 'changes-requested'
    → writes feedback to "UNIFY Review" section on completion record (replace_section)
    → agent re-enters UNIFY as fresh session
    → assembleUnifyContext() detects 'changes-requested' + loads feedback
    → provides revision-aware prompt: "Address this feedback, then set status back to 'pending'"
    → agent addresses feedback, updates record, sets unify_review_status: 'pending'
    → agent calls coda_advance(target_phase: 'done')
    → cycle repeats until human approves
```

### Multi-round Review

The review cycle supports multiple rounds:
```
'pending' → (human approves) → 'approved' → DONE
'pending' → (human requests changes) → 'changes-requested'
         → (agent revises, resets) → 'pending'
         → (human requests changes again) → 'changes-requested'
         → (agent revises, resets) → 'pending'
         → (human approves) → 'approved' → DONE
```

Each round replaces the previous "UNIFY Review" section on the completion record, so the agent always sees only the latest feedback.

## Risk Assessment

- **Low risk:** Changes follow an established pattern (review→build gate) with well-understood code paths
- **Contained blast radius:** Core type changes are additive (optional field), gate change is additive (extra check), tool changes mirror existing handling
- **Backward compatibility:** `unify_review_status` is optional on CompletionRecord, so existing records without it will gate-fail naturally (which is correct — they should require review)
- **Fresh-session safety:** Revision context is loaded from disk (completion record body), not conversation history, which aligns with UNIFY's "fresh autonomous session" strategy
