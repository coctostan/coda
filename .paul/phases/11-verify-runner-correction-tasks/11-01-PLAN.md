<objective>
## Goal
Add the Phase 11 verify runner so VERIFY can autonomously cycle between `verify` and `correct`, emit durable verification-failure artifacts per unmet acceptance criterion, generate correction tasks with unified numbering, and stop cleanly on success or exhaustion.

## Purpose
This turns the second v0.2 autonomous loop into executable workflow behavior by connecting Phase 9 submode/iteration state and Phase 10 disk-backed artifacts to the BUILD loop, without yet pulling in human-review persistence, rewind controls, or Pi runtime session wiring.

## Output
- New `packages/coda/src/workflow/verify-runner.ts` runner with per-AC verification outcomes, failure-artifact writing, correction-task generation, and verify/correct loop handling
- Build/context updates in `build-loop.ts`, `context-builder.ts`, `phase-runner.ts`, `types.ts`, and `index.ts` so correction tasks run through the existing BUILD mechanics with failure context injected
- Minimal core task-frontmatter extension in `packages/core/src/data/types.ts` for correction metadata (`fix_for_ac`)
- Workflow tests covering unmet AC detection, failure artifact format, correction task numbering/fields, correction-task context, verify re-entry, exhaustion, and green-suite exit
</objective>

<acceptance_criteria>
## AC-1: Verify failures are recorded per unmet acceptance criterion
```gherkin
Given a verify-phase issue with task summaries and at least one acceptance criterion that is not satisfied by the built artifacts or test evidence
When the verify runner evaluates the issue in `submode: "verify"`
Then it marks that criterion as not met, writes a verification-failure artifact under `.coda/issues/{slug}/verification-failures/`, and transitions the workflow state toward `submode: "correct"`
```

## AC-2: Correction tasks are generated deterministically with unified numbering
```gherkin
Given one or more verification-failure artifacts for unmet acceptance criteria
When the runner generates correction tasks
Then each correction task defaults to one unmet AC, uses `kind: "correction"`, carries `fix_for_ac` and `covers_ac` for the target AC, uses relevant files from the failure artifact, and continues task ids after the planned task set without reusing prior correction ids
```

## AC-3: Correction work reuses BUILD mechanics with failure-focused context
```gherkin
Given a verify-phase issue in `submode: "correct"` with generated correction tasks
When workflow context is assembled for the active correction task
Then it includes the correction task record, the relevant verification-failure artifact, and source-task summaries while preserving the existing BUILD-loop sequencing and TDD expectations
```

## AC-4: Verify/correct loop exits cleanly on success or exhaustion
```gherkin
Given the verify/correct loop is running with configured `max_verify_iterations`
When all acceptance criteria are eventually met and the full regression suite passes
Then the runner reports a success outcome without generating more correction tasks
And when the loop has already reached its verify-iteration budget
Then it returns an exhausted outcome and preserves the current failure artifacts for later human handling
```

## AC-5: Workflow exports and tests cover Phase 11 behavior without Phase 12+ scope creep
```gherkin
Given the Phase 11 implementation is complete
When the targeted workflow tests, full Bun test suite, and TypeScript typecheck are run
Then they pass without regressions and no human-review gate persistence, rewind controls, or Pi runtime orchestration changes are required for the implementation to work
```
</acceptance_criteria>
