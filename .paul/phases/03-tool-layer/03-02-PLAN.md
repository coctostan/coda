<objective>
## Goal
Implement the lifecycle tools that coordinate phase transitions, runtime status, test execution, and write protection: `coda_advance`, `coda_status`, `coda_run_tests`, and the `write-gate` interceptor.

## Purpose
These tools complete the M3 tool layer. Together with Plan 01's CRUD tools, they form the complete sanctioned API for modifying `.coda/` data and enforcing the development lifecycle. The write-gate is the enforcement mechanism that prevents direct `.coda/` writes and enforces TDD.

## Output
- 4 new source files + 4 test files in `packages/coda/src/tools/`
- Updated types, barrel exports, and `packages/coda/src/index.ts`
- Full M3 tool layer operational
</objective>

<acceptance_criteria>

## AC-1: coda_advance Passes Gates
```gherkin
Given a focused issue at phase "specify" with acceptance criteria
When coda_advance is called with target_phase "plan"
Then the gate passes, issue.phase is updated in mdbase, state.json is updated
And the result includes previous_phase and new_phase
```

## AC-2: coda_advance Blocks on Failed Gate
```gherkin
Given a focused issue at phase "specify" with zero acceptance criteria
When coda_advance is called with target_phase "plan"
Then the gate fails with reason about acceptance criteria
And neither mdbase nor state.json are modified
```

## AC-3: coda_status Returns State Snapshot
```gherkin
Given a CodaState with focus_issue, phase, tasks, and tdd_gate values
When coda_status is called
Then it returns all state fields plus a human-readable next_action
```

## AC-4: coda_run_tests Manages TDD Gate
```gherkin
Given tdd_gate is "locked"
When coda_run_tests runs with mode "tdd" and tests fail (exit ≠ 0)
Then tdd_gate becomes "unlocked" and exit_code is recorded
When tests subsequently pass (exit == 0)
Then tdd_gate becomes "locked"
```

## AC-5: Write-Gate Blocks .coda/ Direct Writes
```gherkin
Given a write/edit request targeting a .coda/ path
When the write-gate checks the request
Then it blocks with reason "Use coda_* tools to modify .coda/ files"
```

## AC-6: Write-Gate Enforces TDD
```gherkin
Given tdd_gate is "locked"
When a write to a non-test file is attempted
Then it blocks with reason about writing a failing test first
When a write to a test file is attempted
Then it allows the write
```
