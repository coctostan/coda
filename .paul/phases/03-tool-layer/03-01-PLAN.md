<objective>
## Goal
Close spec alignment gaps from the Phase 1-2 review, then implement the CRUD tools for `.coda/` records: `coda_create`, `coda_read`, `coda_update`, and `coda_edit_body`. These are the data-manipulation tools that form the first half of the M3 tool layer.

## Purpose
These tools are the only sanctioned write path into `.coda/` — the LLM never writes directly. By implementing CRUD first, we establish the data access pattern that lifecycle tools (Plan 02) will build on.

## Output
- 2 new tests added to state engine (spec alignment)
- `packages/coda/src/tools/` directory with 5 source files and 4 test files
- All tools tested with temp directories, using real M1 (data) layer functions
</objective>

<acceptance_criteria>

## AC-1: Spec Alignment — TDD Gate Cycle
```gherkin
Given a CodaState with tdd_gate "locked"
When the tdd_gate field is set to "unlocked" and the state is persisted
Then loadState returns the state with tdd_gate "unlocked"
And when tdd_gate is set back to "locked" and persisted
Then loadState returns tdd_gate "locked"
```

## AC-2: Spec Alignment — Atomic Write Safety
```gherkin
Given a valid persisted state file
When a .tmp file exists alongside it (simulating a failed write)
Then loadState cleans the .tmp and returns the valid state unchanged
```

## AC-3: coda_create Produces Valid Records
```gherkin
Given a record type and required fields
When coda_create is called
Then a valid mdbase record is created at the correct path
And the returned path and id are correct
And sequential AC IDs are generated for issues
```

## AC-4: coda_read Returns Record Data
```gherkin
Given an existing mdbase record
When coda_read is called with the record path
Then it returns the frontmatter and body
When called with a section parameter
Then it returns only that section's content
```

## AC-5: coda_update Merges Without Clobbering
```gherkin
Given an existing record with frontmatter fields
When coda_update is called with partial field updates
Then only the specified fields change
And the body and unspecified fields remain unchanged
```

## AC-6: coda_edit_body Handles Section Operations
```gherkin
Given an existing record with body sections
When coda_edit_body is called with op "append_section"
Then a new section is added at the end
When called with op "replace_section"
Then only the target section content changes
When called with op "append_text"
Then raw text is appended at the end
```
