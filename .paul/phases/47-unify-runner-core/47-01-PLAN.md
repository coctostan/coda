<objective>
## Goal
Build the UNIFY Runner Core — the workflow orchestrator and supporting infrastructure that enables the 5 mandatory UNIFY actions from the spec. This includes:
1. A `unify-runner.ts` that assembles rich UNIFY context and provides the agent with structured instructions for all 5 actions
2. Expanded UNIFY→DONE gate that enforces completion of spec delta merge, reference doc review, and milestone update
3. Gate data gathering that reads completion record fields to populate the expanded gate

## Purpose
UNIFY is CODA's compounding engine — the spec says "Every completed issue should make future issues easier." Today UNIFY is a one-line system prompt ("Write the completion record and update reference docs") with a gate that only checks `completionRecordExists`. This phase makes UNIFY a first-class workflow phase with structured context, all 5 mandatory actions, and meaningful gate enforcement.

## Key Design Decision
D1 (from milestone): UNIFY is autonomous. Agent writes proposed changes as artifacts on disk.
Human approval is gate-mediated (Phase 48). Phase 47 builds the runner and gate infrastructure —
the gate fields exist but the human approval mechanism is Phase 48's concern. For now, the gate
checks that the completion record reports these fields as true (trusting the agent's report).
Phase 48 will add the review step that validates the agent's work before the record is finalized.

       **`assembleUnifyContext(codaRoot, issueSlug, state?): PhaseContext`**
       This function assembles the full UNIFY context with structured instructions.

       Context assembly:
       a. Load issue record (includes spec_delta if present in frontmatter)
       b. Load latest plan overview
       c. Load ALL completed task summaries (use getPreviousTaskSummaries with full count)
       d. Load topic-matched reference docs: use the issue's `topics` array to call
          `getSectionsByTopics()` on each ref doc in `.coda/reference/` — inject only matching
          sections, not entire documents. If topics array is empty, include full ref-system.md only.
       e. Load module findings summary via loadModuleFindingsSummary
       f. Check if issue has a `milestone` field for action 5 context

       System prompt — must instruct the agent to perform all 5 mandatory actions.
       Use `ACTION 1:` through `ACTION 5:` headers in the prompt string (not markdown ##
       which gets parsed). The reordered action sequence:

       ACTION 1: Merge Spec Delta into ref-system.md
       - Read the spec delta from the issue (if present in frontmatter or body)
       - Compare the planned delta against what was actually built (check task summaries)
       - Produce the FINAL delta — adjusted for any deviations from the original plan
       - Use coda_edit_body to apply ADDED/MODIFIED/REMOVED changes to reference/ref-system
       - If no spec delta exists on the issue, confirm explicitly: no spec change needed
       - Remember whether you made changes (you will report this in the completion record)
       ACTION 2: Review and Update Other Reference Docs
       - Check which reference docs share topics with this issue
       - For each matching doc: does this issue change anything about the documented content?
       - If yes: use coda_edit_body to update the relevant sections
       - If no updates needed: confirm explicitly
       ACTION 3: Capture Knowledge for Compounding
       - Ask: "Would the system catch this automatically next time?"
       - New conventions → update ref-conventions (if it exists) via coda_edit_body
       - New patterns → create or update reference docs
       - New decisions → create decision records via coda_create type 'decision'
       - Lessons learned → update relevant ref doc sections
       ACTION 4: Update Milestone Progress
       - If the issue has a milestone field, check if a milestone record exists
       - If a milestone record exists: update its success criteria status
       - If no milestone field or no milestone record: this action is satisfied automatically
       ACTION 5: Write Completion Record (LAST — after all other actions)
       - Use coda_create with type 'record' to create the completion record
       - Include frontmatter: title, issue slug, completed_at (ISO date), topics (from issue),
         system_spec_updated: true (if you merged changes OR confirmed no change),
         reference_docs_reviewed: true (after completing Action 2),
         milestone_updated: true (after completing Action 4 — even if no milestone exists)
       - Include body sections: Summary, Verification Evidence (per-AC), Deviations,
         Decisions, Patterns, Module Findings
       IMPORTANT: The completion record frontmatter fields (system_spec_updated,
       reference_docs_reviewed, milestone_updated) MUST all be true before you can
       advance to DONE. The gate will check these fields. Since you write the record
       LAST, all fields should reflect the actual outcomes of Actions 1-4.
