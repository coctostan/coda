## Design

### Alignment fixes (lightweight)

**A1: `/coda new` gate-config-aware issue creation**
- When creating an issue, resolve `plan_review` gate mode from config
- If `plan_review` resolves to `auto` for the issue type → set `human_review: false`
- If `plan_review` resolves to `human` or `auto-unless-block` → set `human_review: true`
- Fallback: if no config loaded, keep `human_review: true` (current behavior)
- This is a 5-line change in `commands.ts`

**A2: Review runner gate-aware `human_review_status`**
- After approving the plan in `handleApprovedReview()`, resolve `plan_review` gate mode
- If `auto` → set `human_review_status: 'not-required'` (skip human review)
- If `human` or `auto-unless-block` → keep current behavior (`pending` or `not-required` from issue)
- This is a ~10-line change in `review-runner.ts`
- Depends on `loadCodaConfig()` — already available pattern from `coda-advance.ts`

**A3: `coda-status` gate mode display**
- When phase is `review` and `human_review_required` is shown, also check if gate config overrides it
- Add `gate_mode?: string` to `StatusResult` (optional, informational)
- Populate from `resolveGateMode('plan_review', issueType, config)` when available
- This is informational — the advance handler is still the authority

### E2E tests (the core of Phase 51)

**E1: Full v0.8 lifecycle with gate automation + overlays**
A single test that exercises the full lifecycle specify→done with:
- Gate config: `plan_review: 'auto'`, `build_review: 'auto'`, `unify_review: 'auto'`
- Module overlay present for `security` module
- Walks through: specify→plan→review→build→verify→unify→done
- Verifies: human review auto-skipped, module findings check auto-skipped, UNIFY review auto-approved
- Verifies: overlay content was available to the dispatcher (via prompt assembly check)

**E2: Gate automation with per-issue-type overrides**
- Default gates: `plan_review: 'human'`
- Override: `bugfix: { plan_review: 'auto' }`
- Create a `bugfix` issue → verify review→build advances without human approval
- Create a `feature` issue → verify review→build blocks on human review pending

**E3: Backward compatibility — `human_review_default` without `gates`**
- Config has `human_review_default` but no `gates` field
- Feature with `human_review_default.feature = true` → plan_review resolves to `human`
- Chore with `human_review_default.chore = false` → plan_review resolves to `auto`

**E4: `/coda new` respects gate config (after A1)**
- With `plan_review: 'auto'` in config → new issue gets `human_review: false`
- With `plan_review: 'human'` in config → new issue gets `human_review: true`
- With no config → new issue gets `human_review: true` (fallback)

**E5: Review runner alignment (after A2)**
- With `plan_review: 'auto'` → review runner sets `human_review_status: 'not-required'`
- With `plan_review: 'human'` → review runner sets `human_review_status: 'pending'`
