<objective>
## Goal
Add the Phase 12 human review gate so autonomous REVIEW approval pauses for durable human review when required, human approval can advance cleanly to BUILD, and human-requested changes send the workflow back into revise with persisted review state.

## Purpose
This lands the first human-in-the-loop control on top of the Phase 10 review loop and Phase 11 verification plumbing without broadening into Phase 13 exhaustion/rewind controls or Phase 14 submode-aware Pi runtime orchestration.

## Output
- Review-runner behavior that records `human_review_status` as `pending` instead of treating autonomous approval as sufficient when `issue.human_review === true`
- Tooling/command updates so human approval and human-requested changes persist plan state and route the workflow correctly
- Status/gate behavior that surfaces pending human review and blocks review → build until approval is explicit
- Regression tests covering pending, approved, and changes-requested paths
</objective>

<acceptance_criteria>
## AC-1: Autonomous review approval pauses for required human review
```gherkin
Given a review-phase issue whose plan passes autonomous review and whose issue frontmatter has `human_review: true`
When the review runner completes the review path
Then the plan remains durably marked for human review with `human_review_status: "pending"`
And the workflow does not treat the issue as ready for BUILD until a human explicitly approves it
```

## AC-2: Human approval and change requests persist durable review state
```gherkin
Given a plan whose `human_review_status` is `pending`
When a human approves the plan
Then the plan is updated to `human_review_status: "approved"` and review → build can succeed
And when a human requests changes with feedback
Then the plan body records that feedback, `human_review_status` becomes `"changes-requested"`, and the workflow returns to revise with `loop_iteration` reset for a fresh guided pass
```

## AC-3: Review→build gating and operator guidance respect human review state
```gherkin
Given a review-phase issue that requires human review
When the plan is still `pending` or `changes-requested`
Then review → build is blocked with a human-review-specific reason
And status/command output tells the operator whether to approve the plan or provide review feedback instead of suggesting ordinary build progression
```

## AC-4: Phase 12 remains scoped to human review persistence and gating
```gherkin
Given the Phase 12 implementation is complete
When targeted tests, the full Bun suite, and TypeScript typecheck are run
Then they pass without regressions
And no exhaustion handling, `/coda back` / `/coda kill`, or broader Phase 14 Pi runtime wiring is required for the feature to work
```
