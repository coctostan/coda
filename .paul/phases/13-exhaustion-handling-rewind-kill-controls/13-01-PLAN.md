<objective>
## Goal
Add the Phase 13 exhaustion-handling and operator-control slice so REVIEW/VERIFY loops pause cleanly when their iteration budget is exhausted, `/coda advance` can resume the correct human-mediated path, and new `/coda back` and `/coda kill` commands safely rewind or terminate work without deleting historical artifacts.

## Purpose
This lands the first recovery controls for autonomous loops. It keeps the Phase 10–12 review/verify behavior usable when automation stalls, while preserving institutional knowledge and stopping short of the broader Phase 14 Pi runtime/context-injection work.

## Output
- Exhaustion-aware review/verify runner behavior with actionable pause outcomes for human intervention
- Tooling and command support for manual approval/manual-fix resume, `/coda back <phase>`, and `/coda kill`
- Preservation/supersede semantics for rewind paths instead of destructive cleanup
- Regression tests covering exhaustion, rewind, kill, and operator guidance
</objective>

<acceptance_criteria>
## AC-1: Review and verify exhaustion pause with actionable human paths
```gherkin
Given a review or verify loop whose `loop_iteration` has reached the configured maximum
When the runner is invoked again in that exhausted state
Then it returns an explicit exhausted outcome that preserves the current artifacts/state
And operator-visible guidance identifies the remaining issues or unmet ACs plus the valid human next actions for that phase
```

## AC-2: Exhaustion resume paths are phase-correct and reset loop state cleanly
```gherkin
Given an exhausted review-phase issue
When a human manually approves with `/coda advance`
Then the plan is durably marked approved and the workflow can enter BUILD without pretending the plan passed autonomous review
And given an exhausted verify-phase issue
When a human fixes code and then uses `/coda advance`
Then the workflow re-enters verify with `loop_iteration` reset to `0` instead of incorrectly advancing to UNIFY
```

## AC-3: `/coda back` and `/coda kill` preserve history while resetting active state appropriately
```gherkin
Given an issue in a later lifecycle phase with plan/task artifacts on disk
When `/coda back <phase>` is used for a valid prior phase
Then downstream process state is reset for that target phase without deleting historical files
And plans are marked `superseded` when rewinding to plan/specify while review rewinds keep the active plan editable
And when `/coda kill` is used
Then the issue is marked terminal (`wont-fix` / `done`) and active runtime state is cleared
```

Given the Phase 13 implementation is complete

Then they pass without regressions
And the change stays limited to exhaustion pause flows, manual recovery controls, and rewind/kill semantics without pulling in Phase 14 before-agent-start/context-injection work
