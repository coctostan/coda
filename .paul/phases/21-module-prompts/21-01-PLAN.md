  <name>Task 1: Create security module prompts (2 files)</name>
  <files>modules/prompts/security/pre-plan.md, modules/prompts/security/post-build.md</files>
  <action>
    Create the `modules/prompts/security/` directory and two prompt files.

    **Both files follow the convention from docs/v0.3/04-module-prompts.md:**
    - Header: `## Security — [Hook Point Description]`
    - Opening: `You are checking security aspects of [what].`
    - Sections: `### Context`, `### What to Check`, `### Severity Guide`, `### Assumption Guidance`
    - Do NOT include the JSON output block — the dispatcher appends that automatically via FINDINGS_OUTPUT_TEMPLATE.

    **security/pre-plan.md:**
    - Hook description: "Pre-Plan Security Assessment"
    - Context: You have access to the planned files and issue scope. Evaluate security implications before planning begins.
    - What to Check (4 items from spec):
      1. Files handling authentication or authorization
      2. Files processing user input (injection vectors)
      3. Secret/credential patterns in planned files
      4. Security-sensitive areas in scope (auth, payments, PII)
    - Severity Guide: CRITICAL (hardcoded secrets), HIGH (missing auth, SQL injection risk), MEDIUM (missing input validation, weak passwords), LOW (console.log with sensitive data), INFO (no issues)

    **security/post-build.md:**
    - Hook description: "Post-Build Security Review"
    - Context: You have access to the actual code changes from BUILD. Review the diffs for security concerns.
    - What to Check (5 items from spec):
      1. Hardcoded secrets (API keys, passwords, tokens in source)
      2. Injection risks (eval, exec, innerHTML, raw SQL interpolation)
      3. Auth middleware presence on new endpoints
      4. Input validation on new endpoints (Zod, Joi, etc.)
      5. Sensitive data in logs
    - Severity Guide: same levels as pre-plan but applied to actual code

    Avoid: Including any JSON output formatting instructions — dispatcher handles that. Avoid making prompts too prescriptive about implementation — they should describe WHAT to check, not HOW the agent should code.
  </action>

  <action>
    Create the `modules/prompts/tdd/` directory and three prompt files.

    **All files follow the same convention:**
    - Header: `## TDD — [Hook Point Description]`
    - Opening: `You are checking test-driven development aspects of [what].`
    - Sections: `### Context`, `### What to Check`, `### Severity Guide`, `### Assumption Guidance`
    - Do NOT include JSON output block.

    **tdd/pre-build.md:**
    - Hook description: "Pre-Build TDD Enforcement"
    - Context: You are reviewing the upcoming BUILD task. Ensure TDD discipline is established.
    - What to Check (3 items from spec):
      1. Remind of RED-GREEN-REFACTOR cycle
      2. Explain write-gate mechanics (production writes blocked until failing test)
      3. List anti-patterns: writing production code first, testing after, skipping edge cases
    - This is an upgraded version of the v0.1 `modules/prompts/todd.md` — retain the core TDD enforcement spirit but follow the new v0.3 prompt structure convention. Do not copy-paste the old file; rewrite to the new structure.
    - Severity Guide: HIGH (skipping RED, writing production first), MEDIUM (weak test assertions), LOW (missing edge case tests), INFO (TDD summary)

    **tdd/post-task.md:**
    - Hook description: "Post-Task TDD Compliance Check"
    - Context: A task just completed. Evaluate whether TDD discipline was followed during execution.
    - What to Check (4 items from spec):
      1. Were tests written before production code?
      2. Do tests cover the task's test scenarios from the task record?
      3. Are tests testing behavior (what) or implementation (how)?
      4. Any tautological tests (always pass regardless of implementation)?
    - Severity Guide: HIGH (tests don't cover an AC, tautological test found), MEDIUM (testing implementation instead of behavior), LOW (minor assertion improvements), INFO (task TDD summary)

    **tdd/post-build.md:**
    - Hook description: "Post-Build TDD Summary"
    - Context: All BUILD tasks are complete. Evaluate overall TDD compliance and suggest improvements.
    - What to Check (3 items from spec):
      1. Run full test suite — all tests should pass
      2. Coverage check: are all ACs covered by at least one test?
      3. Suggest REFACTOR candidates from changed files (complexity, duplication)
    - Severity Guide: HIGH (tests don't cover an AC, tautological test), MEDIUM (testing implementation not behavior), LOW (refactor opportunity), INFO (coverage summary)

    Avoid: Including JSON output formatting. Avoid being overly prescriptive about specific test frameworks — keep prompts framework-agnostic.
  </action>
